## Context and Problem Statement

A Project Developer must be able to use AI agents to orchestrate a template upgrade — analysing the change between template versions, applying the upgrade, and validating downstream effects — through an AI-driven workflow that includes review gates, change analysis, and downstream-impact assessment (`cpt-frontx-fr-ai-upgrade-orchestration`, `cpt-frontx-usecase-ai-driven-template-upgrade`). A single change-set engine already owns the production of a reviewable, approvable, non-destructive upgrade change set in the CLI (`cpt-frontx-adr-upgrade-changeset-engine`). How should the AI orchestration relate to that engine — as a coordinating layer above it, or as a second, independent mechanism?

## Decision Drivers

* **One source of truth for the change set** — the bytes applied to a project on upgrade must be produced by one engine, so that an AI-driven upgrade and a direct CLI upgrade yield the same reviewable change set (`cpt-frontx-fr-ai-upgrade-orchestration`).
* **Alongside, not subordinate** — the AI workflow runs alongside direct CLI invocation as a guided path, not as a replacement for or a dependent of the CLI's command surface (`cpt-frontx-usecase-ai-driven-template-upgrade`).
* **Review-gate integrity** — developer review and approval must gate the upgrade before any project file is written; the orchestration adds gates and analysis on top of an engine whose apply step is already non-destructive (`cpt-frontx-fr-cli-upgrade-review-approval`).
* **Recoverable, non-destructive apply** — a version transition must be reversible and must never leave a project half-written, a property already guaranteed by the single change-set engine and inherited rather than reimplemented.
* **AI value-add** — the orchestration must contribute change analysis and downstream-impact assessment that enrich the engine's raw change set, not duplicate the engine's mechanics.

## Considered Options

* **Single CLI engine, AI orchestrates and enriches it** — there is exactly one template-version change-set engine, owned by the CLI (`cpt-frontx-adr-upgrade-changeset-engine`). The AI workflow surface invokes that engine to produce the change set, then enriches the flow with change analysis, review gates, and downstream-impact assessment. It operates alongside the CLI, not subordinate to it, and implements no second engine.
* **Two independent engines** — the AI Tooling Framework implements its own change-set engine for AI-driven upgrades, separate from the CLI's engine, each producing its own change set.
* **AI as a thin pass-through** — the AI workflow merely forwards the developer's intent to the CLI engine and relays its output verbatim, adding no analysis, gates, or downstream-impact assessment of its own.

## Decision Outcome

Chosen option: **Single CLI engine, AI orchestrates and enriches it**, because it is the only option that keeps one source of truth for the applied change set while still delivering the guided, analysis-rich workflow the use case calls for. There is a single template-version change-set engine, and it lives in the CLI (owned by `cpt-frontx-adr-upgrade-changeset-engine`). The AI orchestration layer **invokes** that one engine to produce the reviewable change set and **enriches** the flow around it — change analysis of the version transition, review gates before apply, and downstream-impact assessment — surfacing incompatibilities for the developer to act on before the change set is applied. The orchestration operates **alongside, not subordinate to** the CLI: a developer may upgrade directly through the CLI or through the AI-guided flow, and both apply the identical change set produced by the same engine. The AI layer does **not** implement a second change-set engine.

The scope of this decision is the relationship between the AI orchestration surface and the change-set engine — which side owns engine mechanics (the CLI) and which side owns the guided workflow (the AI Tooling Framework). It does not redefine the engine's internals or its change-set format (those belong to `cpt-frontx-adr-upgrade-changeset-engine`). The two-engines option produces divergent change sets for the same upgrade and doubles the surface that must remain non-destructive and recoverable; the thin-pass-through option keeps a single engine but abandons the analysis, gating, and downstream-impact assessment the use case requires, leaving the AI flow with no value over a bare CLI call.

### Consequences

* Good, because exactly one engine produces the applied change set, so an AI-driven upgrade and a direct CLI upgrade are byte-for-byte equivalent and equally reviewable.
* Good, because the AI flow adds change analysis, review gates, and downstream-impact assessment without duplicating engine mechanics.
* Good, because the engine's non-destructive, recoverable apply and its rollback behaviour are inherited unchanged; the orchestration adds no new way to half-write a project.
* Neutral, because the orchestration depends on the CLI engine's invocation surface remaining stable across versions, which the matched-version policy already governs.
* Bad, because a defect or limitation in the single engine affects both the direct and the AI-guided path; there is no independent second implementation to fall back on.
* Bad, because the orchestration's enrichment is only as good as the engine's change-set fidelity — the AI layer cannot surface a downstream impact the change set does not expose.

### Confirmation

Confirmed by review and by an integration check that the AI-guided upgrade and the equivalent direct CLI upgrade, run against the same project and the same target template version, produce the identical change set — demonstrating a single engine rather than two. A code review confirms the AI Tooling Framework contains no second change-set engine implementation and reaches the engine only through the CLI's invocation surface, and that a review gate stands before any project file is written.

## Pros and Cons of the Options

### Single CLI engine, AI orchestrates and enriches it

One engine in the CLI produces the change set; the AI layer invokes it and enriches the flow with analysis, gates, and downstream-impact assessment, operating alongside the CLI.

* Good, because there is one source of truth for the applied change set across both upgrade paths.
* Good, because review gates and analysis are added without reimplementing engine mechanics.
* Good, because the engine's non-destructive apply and rollback are inherited intact.
* Neutral, because it couples the orchestration to the engine's invocation surface, which the version policy already stabilizes.
* Bad, because both paths share the single engine's limitations.

### Two independent engines

The AI Tooling Framework implements its own change-set engine, separate from the CLI's.

* Good, because the AI path could evolve its engine independently of the CLI's release cadence.
* Bad, because the same upgrade can yield two divergent change sets, undermining a single reviewable result.
* Bad, because the non-destructive, recoverable apply guarantee must now be built and maintained twice.
* Bad, because two engines double the surface a developer must trust and a reviewer must reason about.

### AI as a thin pass-through

The AI workflow forwards intent to the single CLI engine and relays its output verbatim, adding nothing.

* Good, because it preserves a single engine with minimal new surface.
* Bad, because it delivers no change analysis, no review gating, and no downstream-impact assessment, so it fails the use case.
* Bad, because it gives the AI-guided path no value over a direct CLI call, defeating the reason to offer it.

## More Information

The single engine this orchestration coordinates with is decided in `cpt-frontx-adr-upgrade-changeset-engine` (the CLI's change-set engine); the orchestration is delivered by the AI Tooling Framework component `cpt-frontx-component-ai-tooling-kit` and the engine by `cpt-frontx-component-cli`. These pointers identify related decisions and components and are not part of this decision's durable identity.

Integration treatment (INT): the orchestration integrates with one party — the CLI's change-set engine — across the engine's invocation surface. **Integration impact (INT-ADR-001)** — the orchestration consumes the engine's invocation surface and change-set output; their version compatibility is anchored to the ecosystem's matched-version policy rather than negotiated ad hoc, so an upgrade-orchestration release pairs with a compatible engine release. **Contract changes (INT-ADR-002)** — a change to the engine's invocation surface or change-set shape is a contract change that the orchestration must absorb; backward-compatible engine additions leave the orchestration working unchanged, while an incompatible change is governed by the version bump that pairs the two. The version-transition path for the orchestration itself is therefore the matched-version policy, not a separate negotiation.

Reliability treatment (REL): **Reliability impact (REL-ADR-001)** — the orchestration adds no new failure mode to the apply step, because it does not write project files itself; the single engine's non-destructive apply remains the only writer, so a failure during analysis or gating leaves the project untouched. The single engine is a shared dependency of both upgrade paths — a single point of failure for upgrades by design — and the trade-off is accepted in exchange for one source of truth for the change set. **Operational readiness (REL-ADR-002)** — recovery from a declined or failed upgrade is the engine's existing behaviour: no change set is applied, so the project remains at its current version with no files written; the orchestration's review gate is the point at which a developer can decline before any apply, and the downstream-impact assessment is surfaced before the gate so incompatibilities are visible at decision time.

Applicability of the remaining checklist categories: **PERF** — Not applicable, because no latency or throughput budget is bound to the orchestration relationship; the cost of change analysis is a consequence of the workflow, not a target fixed here. **SEC** — Not applicable, because the orchestration introduces no secret material and no new authentication surface beyond what the CLI engine already requires. **DATA** — Not applicable, because the change-set format is owned by `cpt-frontx-adr-upgrade-changeset-engine`, not redefined here (DATA-ADR-NO-001). **OPS** — Not applicable, because there are no standalone operational procedures for the orchestration beyond the engine's existing apply and recovery behaviour. **COMPL** — Not applicable, because no regulatory obligation bears on the orchestration relationship. **ARCH-ADR-008 (supersession)** — Not applicable, because this ADR supersedes no live ADR. **Review cadence**: revisit if a credible need arises for an AI-path engine distinct from the CLI's, or if the engine's invocation surface proves too narrow to support the analysis and downstream-impact assessment the orchestration must provide.
