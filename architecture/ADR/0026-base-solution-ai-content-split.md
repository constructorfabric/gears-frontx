**ID**: `cpt-frontx-adr-base-solution-ai-content-split`
## Context and Problem Statement

The AI Tooling Framework delivers the base AI capabilities — skills, workflows, guidelines, and reference artifacts — that give an AI agent ecosystem fluency in any FrontX-based project (`cpt-frontx-fr-ai-frontx-skills`). The ecosystem also expects template-specific AI capabilities to be available to agents, and a single base framework is intended to serve many independently authored solutions. Where, then, does solution-specific AI content belong: inside the base framework, or somewhere that travels with the solution it describes (`cpt-frontx-fr-ai-tooling-template-agnostic`)?

## Decision Drivers

* **Template-agnostic guarantee** — the base framework must serve any template without carrying knowledge of any particular one, so it stays portable across every template (`cpt-frontx-fr-ai-tooling-template-agnostic`).
* **Separation of concerns** — ecosystem-wide AI capability and solution-specific AI capability are distinct responsibilities and must not be entangled in one corpus.
* **Single-base-many-solutions reuse** — one base framework is expected to back many solutions at once; baking any solution's content into the base would force every other solution to carry knowledge it does not use.
* **Blast-radius isolation** — a change to one solution's AI content must not perturb the base framework or any unrelated solution; the partition bounds where such a change can take effect.
* **Single source of solution capability** — each solution's AI capability must have exactly one home, so agents are not assisted by stale or duplicated solution content.

## Considered Options

* **Solution-agnostic base; solution content in the template's AI bundle** — the base framework carries only ecosystem-wide AI capability and zero solution-specific content. Each solution's AI content is carried by that solution's template, inside the template's AI-extension bundle (the bundle whose shape is fixed by `cpt-frontx-adr-template-ai-extension-contract`), and is admitted as a downstream capability when the template is installed.
* **Single combined corpus** — the base framework carries both ecosystem-wide AI capability and the AI content for every known solution in one shared corpus, partitioned only by internal convention.
* **Per-solution base variants** — a separate build of the base framework is produced for each solution, each variant bundling that solution's AI content alongside the ecosystem-wide capability.

## Decision Outcome

Chosen option: **Solution-agnostic base; solution content in the template's AI bundle**, because it is the only option that honours the template-agnostic guarantee while still giving every solution a single, owned home for its AI capability. The base framework's AI tooling carries zero solution-specific content; a solution's AI content lives in that solution's template AI bundle and reaches agents through the template-extension contract (`cpt-frontx-adr-template-ai-extension-contract`) and the discovery-and-activation mechanism (`cpt-frontx-adr-extension-discovery-activation`). This partition parallels the CLI's template independence (CLI-1, `cpt-frontx-constraint-cli-template-independence`): just as the CLI carries no dependency on any template, the base AI framework carries no dependency on any solution.

The scope of this decision is the placement boundary between solution-agnostic and solution-specific AI content. It does not decide the shape of a template's AI bundle (that is `cpt-frontx-adr-template-ai-extension-contract`), nor how a bundle is found and activated once a template is installed (that is `cpt-frontx-adr-extension-discovery-activation`), nor how the base framework is packaged and delivered (that is `cpt-frontx-adr-kit-packaging-cyber-pilot-kit-frontx`). The combined-corpus option couples the base to every solution it has ever served, breaking the agnostic guarantee and forcing every consumer to carry unrelated solution content; the per-variant option preserves agnosticism within each build but multiplies the base into one artifact per solution, so the single-base-many-solutions reuse is lost and a base-level change must be re-applied to every variant.

### Consequences

* Good, because the base framework remains portable across every solution: it carries no knowledge of any particular one, satisfying the template-agnostic guarantee.
* Good, because each solution has exactly one home for its AI capability — its own template bundle — so agents are never assisted by duplicated or stale solution content.
* Good, because a change to one solution's AI content is bounded to that solution's bundle and cannot perturb the base or any unrelated solution.
* Bad, because an agent only gains a solution's AI capability once that solution's template is installed; out of the box the base provides ecosystem fluency only.
* Bad, because the partition must be enforced — without a check, solution-specific content could accrete into the base over time and silently erode the guarantee.

### Confirmation

Confirmed by review and by an automated admission check in continuous integration: the base framework's AI corpus is scanned for solution-specific content, and any solution-named resource present in the base fails the check (the same class of template-agnostic / `frontx_`-prefixed admission rule used for kit-resource packaging, KIT-1 `cpt-frontx-constraint-kit-prefixed-resource-ids`). A code review of any base-framework change additionally confirms that no solution-specific content has been introduced into the base, and that newly added solution content is carried in the relevant template bundle instead.

## Pros and Cons of the Options

### Solution-agnostic base; solution content in the template's AI bundle

The base framework carries only ecosystem-wide AI capability; each solution's AI content is carried by that solution's template bundle and admitted downstream through the extension contract.

* Good, because the base stays portable across every solution and depends on none.
* Good, because each solution owns its AI capability in a single place that travels with it.
* Good, because the blast radius of a solution-content change is bounded to that solution's bundle.
* Neutral, because it relies on the separately decided extension contract and discovery mechanism to deliver solution capability to agents.
* Bad, because solution capability is available to an agent only after the corresponding template is installed.

### Single combined corpus

The base framework carries ecosystem-wide capability and every known solution's AI content in one corpus, partitioned by internal convention only.

* Good, because all AI content is delivered in one artifact with nothing further to install for known solutions.
* Bad, because the base becomes coupled to every solution it has served, breaking the template-agnostic guarantee.
* Bad, because every consumer carries AI content for solutions it does not use.
* Bad, because a change to any one solution's content rides inside the shared base, so its blast radius is the whole framework.

### Per-solution base variants

A separate base build is produced per solution, each bundling that solution's AI content with the ecosystem-wide capability.

* Good, because each individual variant is internally agnostic to other solutions' content.
* Bad, because the single-base-many-solutions reuse is lost: the base multiplies into one artifact per solution.
* Bad, because an ecosystem-wide capability change must be re-applied and re-released across every variant.

## More Information

The base AI corpus this partition organizes is the ecosystem-wide capability set — skills, workflows, guidelines, and reference artifacts — delivered through the Cypilot kit packaging decided in `cpt-frontx-adr-kit-packaging-cyber-pilot-kit-frontx`. The shape a solution's AI bundle conforms to is decided in `cpt-frontx-adr-template-ai-extension-contract`, and the mechanism that finds and activates such a bundle is decided in `cpt-frontx-adr-extension-discovery-activation`; these are non-binding pointers to related decisions and are not part of this decision's durable identity. The parallel to CLI-1 (`cpt-frontx-constraint-cli-template-independence`) is intrinsic: both the CLI and the base AI framework are solution-agnostic by construction, with solution specifics resolved from the template.
